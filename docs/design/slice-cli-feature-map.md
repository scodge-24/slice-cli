# slice-cli — feature map (as-is)

A single reference that inventories everything the `slice` CLI does today: every command, every
build feature, the data model underneath, and a pointer to the benchmark evidence for each. Written
2026-06-06. It is a **snapshot for reasoning about the surface** (notably the feature-overlap question)
— not a spec and not a roadmap. The roadmap/rationale lives in
[`agentic-search-incorporation-plan.md`](../../design/agentic-search-incorporation-plan.md); the evidence lives in
the sibling repo's [`FINDINGS.md`](../../../slice-cli-benchmark/FINDINGS.md) (line numbers below are a
2026-06-06 snapshot of that file and will drift).

## What the tool is

slice-cli turns a repo that's been carved into per-area **slice cards** (`slices/*.md`, generated by
the `/slice-codebase` skill) into a query surface for humans and agents. A card is a structured doc —
frontmatter (`slice_id`, `description`, `files`, `abstractions`, `dependencies`) + body sections
(`## System Behavior`, `## Invariants`, `## Runtime Flows`, `## Verification`, `## Update Triggers`).
On top of those cards the CLI answers the questions you ask when you land in unfamiliar code (what owns
this file, what breaks if I change it, where does this concept live, how do requests flow), plus a
doc-staleness half that flags design docs whose code has moved.

Two surfaces consume it: **humans** (rich CLI, pick what you want) and **agents** (the benchmark
exposes the commands as function-calling tools). That distinction matters for the overlap discussion
in the last section.

## Commands, grouped by the job they do

Status legend: **shipped** = in the default binary · **gated:ast** / **gated:semantic** = only with that
Cargo feature (see Build features).

### Discover — get your bearings

| command | what it does | status |
|---|---|---|
| `list` | every slice: id, description, LoC. The cheapest inventory. | shipped |
| `browse [-q query] [--print]` | interactive fzf picker with preview lenses (`ctrl-o` card, `ctrl-r` call-stacks, `ctrl-d` verification, `ctrl-t` reverse-deps). Needs fzf ≥ 0.30. | shipped |

### Locate — find code by concept

| command | what it does | status |
|---|---|---|
| `find <needle>` | keyword search over slice descriptions, abstractions, and bodies; tags each hit with where it matched (`[abstractions]`, `[body]`). Multi-term → reports which terms hit nothing. | shipped |
| `find <q> --semantic --units <cards\|code\|code-sliceaug>` | embedding search instead of keyword. `cards` = description+abstractions; `code` = source symbol bodies (each hit carries `file:line`); `code-sliceaug` = code chunks prepended with slice context. Vector score generates candidates; deterministic topology + freshness rank; stale hits flagged. | gated:semantic |
| `grep <slice> <pattern> [-i] [-F] [--symbols]` | ripgrep scoped to one slice's files. `--symbols` annotates each hit with its enclosing function/class span (heuristic, or AST under `ast`). Needs `rg`. | shipped |

> **This is the crowded cluster** — `find` (keyword/cards), `find --semantic` (embedding/code), and
> `grep` (keyword/code) all answer "where is the code for X." See *Feature overlap* below.

### Orient / relate — a file's or slice's surroundings

| command | what it does | status |
|---|---|---|
| `context <file\|slice> [--strict] [--best-effort]` | owning slice + files + dependencies + runtime flows for a path/area. Emits one-line `depends-on:` and `blast-radius:` summaries pointing at the next command (Stage 1). | shipped |
| `for <path>` | just the owning slice id for a file (minimal `context`). | shipped |
| `files <slice>` | the files a slice owns. | shipped |
| `show <slice> [--body\|--system\|--call-stacks\|--verification]` | slice metadata + linked docs; lenses for the body sections; abstractions annotated with `file:line` def sites. | shipped |
| `deps <slice> [--reverse] [--transitive] [--files]` | dependencies (forward) or blast radius (`--reverse`); `--transitive` for the full chain; `--files` turns the radius into concrete collaborator files, nearest-first, deduped (Stage 1 bidirectional). | shipped |

> `context` is nearly a superset of `for` + `files` + a deps summary; `show` overlaps it on the body
> lenses. Five tools, one neighbourhood.

### Structure — symbols within a file/slice

| command | what it does | status |
|---|---|---|
| `outline <file>` | every definition + 1-based line span, in order, plus a `coverage: N/M definitions spanned` line declaring how many the spanner could confidently bound (Stage 2). | shipped |
| `symbols <slice>` | the same across every file a slice owns, with coverage aggregated over the slice (Stage 2). | shipped |

### Doc-staleness — the other half

| command | what it does | status |
|---|---|---|
| `docs <slice>` | design docs linked to a slice + their stale/current status. | shipped |
| `affected-docs <files…>` | docs a set of changed files may have made stale (the post-edit check). | shipped |
| `stale-docs` | every currently-stale doc; exit 1 if any (doubles as a CI gate). | shipped |
| `stamp <doc> \| --slice <id> \| --all` | mark a doc verified against current code — records the content fingerprint. | shipped |
| `docs-bootstrap <dir> [--dry-run] [--force]` | scan a docs directory and generate `DOCS.yaml` from `tracks:` frontmatter (or a commented stub). | shipped |

### Index maintenance

| command | what it does | status |
|---|---|---|
| `sync-index [--stdout] [--check]` | regenerate `INDEX.md` from frontmatter + the source fingerprint. `--check` is a CI gate. | shipped |
| `check [--strict-index] [--no-staleness] [--no-staged-coverage] [--no-doc-drift] [--require-verification]` | integrity checks — index drift, doc staleness, staged coverage, verification links. | shipped |
| `semantic-index --units <cards\|code\|code-sliceaug> [--model] [--dimensions]` | build the embedding index (`slices/SEMANTIC[-code…].json`). Needs `OPENROUTER_API_KEY`. Regenerable, gitignored. | gated:semantic |

## Build features (Cargo)

Both are **off by default** so the shipped binary stays zero-heavy-deps (invariant #4). Each is a
toggle so its cost is opt-in.

| feature | what it adds | deps | status & evidence |
|---|---|---|---|
| `ast` (Stage 3) | swaps a Tree-sitter backend in behind `symbols.rs`'s span queries (`enclosing_span`, `definition_span`, `enumerate_symbols`) for Python — spans the decorated / multi-line / nested defs the heuristic refuses. Runtime escape hatch `SLICE_SYMBOLS=heuristic`. Improves `outline`/`symbols`/`grep --symbols`/`show` anchors. | tree-sitter, tree-sitter-python | **Built, retrieval-neutral.** Declared-coverage jumps (flask `outline` 40/71 → 66/66) but the agent-loop A/B vs heuristic is a wash with a faint negative lean — the binding deficit was never span quality. `FINDINGS.md` §"Stage 3 — AST backend" (L388). Plan §6 (L262). |
| `semantic` (Stage 4) | embedding retrieval: `find --semantic` + `semantic-index`. `Embedder` trait (OpenRouter first, local-able later), default `google/gemini-embedding-2` @512-dim, deterministic rerank, freshness-flagged `SEMANTIC*.json`. | ureq | **Built, gated, mid-iteration.** Strong *retriever* in isolation (+0.21 file-recall@3 on 31 real issues) but REGRESSES the agent loop (−0.13 → −0.08 file coverage after one affordance iteration). `FINDINGS.md` §§"Stage 4 …" (L429–L604). Plan §7 (L329). |

## The agentic-search stages (how features map to the research arc)

The plan ([`agentic-search-incorporation-plan.md`](../../design/agentic-search-incorporation-plan.md)) drives the
recent feature work. Status as of 2026-06-06:

| stage | surface | status | verdict (evidence) |
|---|---|---|---|
| **1 — Dependency anchoring (bidirectional)** | `deps --files`, `context` depends-on/blast-radius | shipped | Shipped + tested; the lever is there but the agent under-routes to it. `FINDINGS.md` §"Bidirectional fix" (L97). Plan §4 (L93). |
| **2 — Symbol orientation** | `outline`, `symbols` | shipped | **Validated** — real coverage + AUC-nav-tax lift under flash+thinking. The one stage that moved the agent metric. `FINDINGS.md` §"flash + THINKING" (L301). Plan §5 (L207). |
| **3 — AST backend** | `ast` feature | gated | Built; **retrieval-neutral** in the loop. `FINDINGS.md` L388. Plan §6. |
| **4 — Semantic / hybrid lens** | `semantic` feature | gated | Built; strong retriever, **regresses the agent loop**; iteration recovered 2/3 instances, xarray stubborn. `FINDINGS.md` L429–604. Plan §7. |
| **5 — MCP wrapper** | — | not built | A candidate surface for the routing work, low priority. Plan §8 (L431). |

Cross-cutting principle **(P) — affordance / ACI:** tool *output* should advertise the next cheap action
(anchored hits, reverse-dep counts, "N collaborator files in the blast radius"). This, not a new
retrieval modality, is repeatedly the actual lever. Plan §8 (L431).

## Data model & subsystems (what the commands sit on)

| artifact / module | role |
|---|---|
| `slices/*.md` (cards) | the substrate: frontmatter (id, description, files, abstractions, dependencies, exclusions) + body (System Behavior, Invariants, Runtime Flows, Verification, Update Triggers). |
| `slices/DOCS.yaml` | single source of truth for doc→slice mapping + staleness; anchored on a content `fingerprint` per doc, with `verified_at` as a human HEAD note. Written by `docs-bootstrap`/`stamp`. |
| `slices/INDEX.md` | generated slice index + source fingerprint (`sync-index`). |
| `slices/SEMANTIC[-code\|-code-sliceaug].json` | embedding index (gated:semantic): provider/model/dims + per-unit `{slice_id, kind, anchor, text, slice_fp, vector}`. Regenerable, gitignored. |
| `src/symbols.rs` | heuristic, zero-dep span detection (reject-on-ambiguity); the `ast` backend in `src/symbols_ast.rs` swaps in behind its three functions. |
| content fingerprint (`commands::content_fingerprint`) | sha-based staleness anchor shared by DOCS.yaml, INDEX.md, and the semantic index's per-slice `slice_fp`. |

**What the binary writes:** only slice-owned state (`DOCS.yaml`, `INDEX.md`, `SEMANTIC*.json`). Host/
project policy (agent instructions, hooks, CI, `.claude/`) is never written by the binary — it's setup
guidance in `docs/setup.md`.

**User docs:** `docs/navigating.md` (human nav guide — mirrors the agent briefing), `architecture.md`,
`manifest-schema.md`, `agent-workflows.md`, `verification-links.md`, `setup.md`.

## Feature overlap & purpose (the open question)

Read against the agent surface, the ~18 commands collapse into **four jobs**, and two of the four are
crowded — which is the live concern (see the 2026-06-06 discussion in this session and the agent-loop
findings):

| job | tools that answer it | overlap risk |
|---|---|---|
| **Locate code** | `find`, `find --semantic`, `grep` (+ baseline `search_text`) | **High.** 3–4 tools, one question. Differ only by *mechanism* (keyword vs embedding) and *corpus* (cards vs code), not by the question asked. |
| **Orient / relate** | `context`, `for`, `files`, `show`, `deps` | **Medium.** `context` ≈ superset of `for`+`files`+deps-summary; `show` overlaps on body lenses. |
| **Structure** | `outline`, `symbols` (+ `grep --symbols`) | Low–medium. file vs slice scope. |
| **Freshness** | `docs`, `affected-docs`, `stale-docs`, `stamp`, `docs-bootstrap` | Low — distinct lifecycle steps. |

**The pattern the evidence shows.** A feature earns its place by answering a question *nothing else
answers*. Stages 1–2 did (blast radius; symbol orientation — and Stage 2 is the one that moved the
agent metric). Stages 3–4 added *mechanisms* (AST spans, embeddings) for questions that already had
answers, so they overlap by construction — and the agent-loop tests show the cost: AST is neutral, and
`slice_semantic` actively **regressed** retrieval by *displacing* the breadth tools (`slice_context`
31→21, `slice_deps` 8→4) it competed with. `FINDINGS.md` §"agent-loop test" (L552) and the iteration
that followed (L591+).

**Implication for the surface (not yet acted on):** the modalities (keyword / AST / embedding) likely
belong *under* a small set of purpose-tools as internal strategies, not as peer tools the agent must
route between — e.g. semantic folded into `find` (cards-keyword first, code-embedding fallback) rather
than a separate `slice_semantic`. AST already follows this shape (invisible, behind `symbols.rs`); the
benchmark's 1:1 CLI→tool exposure is what re-introduces the overlap on the agent side. The human CLI can
keep the rich surface; the agent wants the consolidated one. This is a candidate next direction, gated
on confirming the consolidation removes the regression.
